# 1. 환경 개요 및 아키텍처

이 문서는 **Azure CycleCloud**의 주요 개념, 시스템 아키텍처, 그리고 현재 **KT CycleCloud 환경의 구축 현황 및 운영 지침**을 정리한 가이드입니다.

---

## 1.1 Azure CycleCloud 란?

**Azure CycleCloud**는 Azure 환경에서 HPC(고성능 컴퓨팅) 클러스터를 손쉽게 생성, 관리 및 오케스트레이션(자동 증설/감설)할 수 있도록 지원하는 **엔터프라이즈 관리 서비스**입니다.

- **다양한 스케줄러 지원**: Slurm, OpenPBS, LSF, HTCondor, Grid Engine 등
- **오토스케일링(Autoscale)**: 작업(Job) 큐의 수요에 따라 VMSS 기반 계산 노드를 자동으로 증설 및 감설
- **통합 제어 인터페이스**: 웹 포털 GUI, CLI(`cyclecloud`), REST API 제공

## 1.2 시스템 구성 아키텍처

```
                    ┌─────────────────────────────────────────┐
   운영자/브라우저 ─▶│  CycleCloud 서버 VM (cc-server)          │
   (HTTPS 443)      │   - 웹 포털 / cycle_server 서비스         │
                    │   - cyclecloud CLI                       │
                    │   - System-assigned Managed Identity     │──▶ 구독 범위 노드 생성
                    └───────────────┬─────────────────────────┘   (Contributor)
                                    │ 클러스터 프로비저닝 (CLI / Template)
                     ┌──────────────┼───────────────────────┐
                     ▼              ▼                       ▼
              스케줄러 노드     실행(Execute) 노드 …    (Autoscale)
              (slurm-master)  (VMSS 기반 계산 노드)
                     └──────── VNet: cc-vnet / subnet: compute ──────┘

   Storage Account (cclkekwphusd3i) = "Locker" : 프로젝트/템플릿/cluster-init 객체 저장소
```

### 주요 구성 요소
1. **Control Plane (CycleCloud Server VM)**:
   - CycleCloud 어플리케이션(`cycle_server`)이 구동되는 VM (`cc-server`)
   - Azure 관리 ID(Managed Identity)를 활용하여 시크릿/암호 없이 Azure 리소스 프로비저닝
2. **Scheduler Node (Master Node)**:
   - 클러스터의 워크로드를 관리하고 스케줄링(`slurmctld`)
   - 모든 계산 노드가 공유하는 NFS(`/shared`, `/sched`) 제공
3. **Execute Nodes (Worker Nodes)**:
   - 실제 작업(Job)이 수행되는 계산 전용 VM
   - 작업 제출 시 자동 생성되며 유휴 시 자동 종료
4. **Locker (Blob Storage)**:
   - 클러스터 구성 템플릿, `cluster-init` 커스텀 스크립트 보관 저장소

## 1.6 네트워크·보안 아키텍처 (개념)

> 💡 이 절은 **"왜 이런 구조인가"** 를 설명하는 개념 파트입니다. 실제 클릭/설정 단계는 각 실습 챕터(3·5·2·11장)에 있으며, 아래에서 교차참조합니다.

### 1.6.1 서브넷 구조 — 서버와 계산 노드의 분리

CycleCloud 환경은 하나의 VNet(`cc-vnet`, `10.0.0.0/16`) 안에서 **역할별로 서브넷을 분리**합니다.

| 서브넷 | 대역 | 용도 | 분리 이유 |
|--------|------|------|-----------|
| **서버 서브넷** (`cyclecloud`) | `10.0.0.0/24` | CycleCloud Server VM 전용 | 제어 평면(Control Plane). 포털/오케스트레이션만 담당하므로 NSG를 좁게(443/22) 유지 |
| **계산 서브넷** (`compute`) | `10.0.1.0/24` | 스케줄러·계산 노드(VMSS) 전용 | 데이터 평면(Data Plane). 노드가 **대량·수시로 생성/삭제**되므로 서버와 분리해 NSG·라우팅을 독립 관리 |
| **AzureBastionSubnet** | `/26` 이상 | Bastion 전용 (선택) | Bastion 사용 시 **이름·크기 고정 규격**이 강제됨(`/26` 이상, 이름 변경 불가) |

**IP 사이징 주의 (대규모 운영 핵심)**
- 계산 노드는 **노드 1대당 IP 1개**를 계산 서브넷에서 소모합니다. `/24`(약 251개 가용)로는 대략 **250노드**가 한계입니다.
- KT 운영(Slurm 100노드)처럼 규모가 크거나 향후 증설 여지가 있으면 계산 서브넷을 **`/23`·`/22`로 넉넉히** 설계하십시오. 서브넷은 생성 후 대역 확장이 번거로우므로 **초기 설계 단계에서 여유 확보**가 중요합니다.
- 👉 클러스터 생성 시 서브넷 선택 위치는 [3장 §3.3 Required Settings](03-신규-클러스터-생성.md)를 참고하세요.

### 1.6.2 Private Endpoint & 보안 (스토리지/DB)

보안 강화 환경에서는 스토리지와 DB를 **공용 인터넷 경로 대신 VNet 내부 사설 IP로만** 접근하도록 구성합니다.

- **스토리지(Locker) Private Endpoint**: 스토리지 계정에 **Shared Key(계정 키)를 비활성화**하는 정책을 적용하면(CycleCloud 8.7+ 권장), 노드/서버는 **Managed Identity 로만** Blob에 접근합니다. 이때 공용 엔드포인트를 차단하고 **Private Endpoint(예: `cc-blob-pe`, `10.0.0.5`) + Private DNS Zone**(`privatelink.blob.core.windows.net`)을 구성해 VNet 내부에서 이름 해석이 사설 IP로 되도록 합니다.
  - ⚠️ PE만 만들고 **Private DNS 연결을 빠뜨리면** 노드가 여전히 공용 IP로 이름 해석 → 접근 실패(다운로드 403/타임아웃)합니다. **PE + Private DNS 는 세트**입니다.
  - 이 정책과 MI 권한 부여의 연결 고리는 [3장 신규 클러스터 생성](03-신규-클러스터-생성.md)의 Managed Identity/RBAC 설명을 함께 보세요.
- **DB(Job Accounting) Private Access**: MySQL Flexible Server는 **Private Access (VNet Integration)** 로 공용 IP를 차단하고 VNet에서만 접근합니다. 설정은 [7장 §7.1](07-Job-Accounting-설정.md)을 참고하세요.

### 1.6.3 Public IP & 접근 모델

접근 경로는 **보안 요구 수준에 따라** 선택합니다. 두 가지 서로 다른 Public IP 개념을 구분하세요.

| 구분 | 대상 | 역할 | 트레이드오프 |
|------|------|------|--------------|
| **서버 공인 IP** (`cc-pip`) | CycleCloud Server | 웹 포털(HTTPS 443) 접근 | 편리하지만 포털이 인터넷에 노출 → NSG로 출발지 IP 제한 권장. 운영 환경은 **PE/사설 + Bastion** 로 대체 가능 |
| **Scheduler Public IP** (클러스터 옵션) | 스케줄러 노드 | 포털을 거치지 않고 노드에 직접 SSH | 편의를 위해 계산 노드에 공용 IP 부여 → **미사용(Unchecked) 권장**. 대신 서버를 점프호스트로 사용 |

**Public IP 미사용(사설 전용) 시 접근 방법**
- **Azure Bastion**: 공용 IP 없이 포털에서 브라우저로 SSH/RDP. **현재 랩 환경이 Bastion 방식**입니다.
- **점프호스트(서버 경유)**: CycleCloud 서버에 먼저 접속한 뒤 서버 내장 키로 계산 노드에 SSH. 노드 트러블슈팅의 표준 경로이며 [11장 §11.x](11-트러블슈팅-로그.md)를 참고하세요.

**보안 등급별 권장 구성 (참고)**

| 등급 | 서버 접근 | 노드 접근 | 스토리지 |
|------|-----------|-----------|----------|
| **개발/학습** | 공인 IP + NSG 출발지 제한 | Scheduler Public IP 허용 가능 | Shared Key 허용 가능 |
| **운영(권장)** | 사설 + **Bastion** | 서버 점프호스트만 | **Shared Key 비활성화 + PE + MI** |

> 📌 **요약**: 서브넷은 역할별 분리 + 넉넉한 IP 사이징, 스토리지/DB는 Shared Key 비활성화 시 **PE+Private DNS+MI** 세트, 접근은 운영일수록 **공인 IP 제거 + Bastion/점프호스트** 로 좁힙니다.
