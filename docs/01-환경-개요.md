# 1. 환경 개요 및 아키텍처

이 문서는 **Azure CycleCloud**의 주요 개념, 시스템 아키텍처, 그리고 현재 **KT CycleCloud 환경의 구축 현황 및 운영 지침**을 정리한 가이드입니다.

---

## 1.1 Azure CycleCloud 란?

**Azure CycleCloud**는 Azure 환경에서 HPC(고성능 컴퓨팅) 클러스터를 손쉽게 생성, 관리 및 오케스트레이션(자동 증설/감설)할 수 있도록 지원하는 **엔터프라이즈 관리 서비스**입니다.

- **다양한 스케줄러 지원**: Slurm, OpenPBS, LSF, HTCondor, Grid Engine 등
- **오토스케일링(Autoscale)**: 작업(Job) 큐의 수요에 따라 VMSS 기반 계산 노드를 자동으로 증설 및 감설
- **통합 제어 인터페이스**: 웹 포털 GUI, CLI(`cyclecloud`), REST API 제공

## 1.2 시스템 구성 아키텍처

```
                    ┌─────────────────────────────────────────┐
   운영자/브라우저 ─▶│  CycleCloud 서버 VM (cc-server)          │
   (HTTPS 443)      │   - 웹 포털 / cycle_server 서비스         │
                    │   - cyclecloud CLI                       │
                    │   - System-assigned Managed Identity     │──▶ 구독 범위 노드 생성
                    └───────────────┬─────────────────────────┘   (Contributor)
                                    │ 클러스터 프로비저닝 (CLI / Template)
                     ┌──────────────┼───────────────────────┐
                     ▼              ▼                       ▼
              스케줄러 노드     실행(Execute) 노드 …    (Autoscale)
              (slurm-master)  (VMSS 기반 계산 노드)
                     └──────── VNet: cc-vnet / subnet: compute ──────┘

   Storage Account (cclkekwphusd3i) = "Locker" : 프로젝트/템플릿/cluster-init 객체 저장소
```

### 주요 구성 요소
1. **Control Plane (CycleCloud Server VM)**:
   - CycleCloud 어플리케이션(`cycle_server`)이 구동되는 VM (`cc-server`)
   - Azure 관리 ID(Managed Identity)를 활용하여 시크릿/암호 없이 Azure 리소스 프로비저닝
2. **Scheduler Node (Master Node)**:
   - 클러스터의 워크로드를 관리하고 스케줄링(`slurmctld`)
   - 모든 계산 노드가 공유하는 NFS(`/shared`, `/sched`) 제공
3. **Execute Nodes (Worker Nodes)**:
   - 실제 작업(Job)이 수행되는 계산 전용 VM
   - 작업 제출 시 자동 생성되며 유휴 시 자동 종료
4. **Locker (Blob Storage)**:
   - 클러스터 구성 템플릿, `cluster-init` 커스텀 스크립트 보관 저장소

## 1.3 KT CycleCloud 구축 현황 및 운영 지침 (2026년 기준)

> 💡 **KT 운영 환경 핵심 정리**

| 구분 | 주요 내용 및 규정 |
|------|-------------------|
| **리전 통합 이관** | 기존 UAEN, KRC, KRS 3개 리전에서 **Mexico Region**으로 모두 통합 이관 완료 (운영 Slurm 100노드, 개발 single VM 4개) |
| **재시작 최소화 원칙** | CycleCloud 지원 시 **노드 재시작을 최소화**해야 함. (GPU VM Capa 부족으로 재할당 실패 위험 존재. 불가피 시 Capa 팀 사전 조율 필수) |
| **Reserved Instance (RI)** | KT GPU VM들은 RI가 적용되어 있으므로 의도치 않은 Scale-in을 방지해야 함 (`SuspendExcParts=hpc` 설정) |
| **script 정책** | `cloud-init` 수정 시 전체 클러스터 재기동이 필요하므로 **`cluster-init` 사용 필수** (Locker Blob 스크립트 관리) |
| **Job Accounting** | **Azure Database for MySQL - Flexible Server** 연동으로 `sacct` 작업 이력 영구 보존 |
| **사용자 관리** | **Built-in** 사용자 방식 사용 중. SSH Keypair 등록으로 노드 접근 및 sudo 권한 제어 |
| **스토리지 마운트** | OS Disk 1TB, Blob Storage NFS 2개, Azure Managed Lustre 512TB 공유 마운트 |
| **GPU 모니터링** | Azure Managed Prometheus + Managed Grafana 구성 (Mexico 리전 제약 고려 필요) |

## 1.4 이번 실습 환경에 배포된 리소스

리소스 그룹: `rg-cyclecloud-training` (Korea Central / 한국 중부)

| 리소스 | 이름 | 설명 및 사양 |
|--------|------|--------------|
| **서버 VM** | `cc-server` | CycleCloud 8 Server (Standard_D4s_v5) |
| **Managed Identity** | `cc-server` 시스템 할당 ID | 구독 범위 `Contributor` 역할 (노드 자동 생성 권한) |
| **VNet** | `cc-vnet` | `10.0.0.0/16` |
| **Subnet (서버)** | `cyclecloud` | `10.0.0.0/24` (CycleCloud Server 전용) |
| **Subnet (노드)** | `compute` | `10.0.1.0/24` (HPC 계산 노드 전용) |
| **NSG** | `cc-nsg` | 포털(HTTPS 443), HTTP(80), SSH(22) 허용 |
| **Public IP** | `cc-pip` | DNS 라벨 자동 생성 (`cc-cyclecloud-*.koreacentral.cloudapp.azure.com`) |
| **Storage Account** | `cclkekwphusd3i` | Locker 전용 스토리지 (컨테이너 `cyclecloud`) |

## 1.5 접속 방법 및 권한 구조

### 접속 방법 요약
- **웹 포털 GUI**: `https://cc-cyclecloud-ekwphu.koreacentral.cloudapp.azure.com`
- **서버 CLI 및 로그 확인 (SSH 키 불필요)**:
  - Azure Portal → `cc-server` → **운영 → "명령 실행 (Run Command)"**
  - 또는 Azure CLI: `az vm run-command invoke -g rg-cyclecloud-training -n cc-server --command-id RunShellScript --scripts "tail -50 /opt/cycle_server/logs/cycle_server.log"`
- **SSH 접속**: `ssh -i keys/cyclecloud_rsa azureadmin@<SERVER_PUBLIC_IP>`
