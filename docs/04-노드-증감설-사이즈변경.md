# 4. 노드 증설/감설 및 노드 사이즈 변경

수동/자동 노드 조정 방법과 **KT 환경의 Scale-in 방지 특이사항**을 정리합니다.

---

## 4.1 노드 증감설 방식 비교

CycleCloud 노드는 **노드 배열(node array)** 단위로 관리됩니다(예: `scheduler`, `hpc`, `htc`).

| 방식 | 동작 | 사용 상황 |
|------|------|-----------|
| **자동확장 (Autoscale)** | Slurm 큐 수요에 따라 노드 자동 생성·유휴 시 자동 축소 | 일반 운영 권장 |
| **수동 (Manual)** | CLI(`azslurm`)/포털에서 노드 수 직접 제어 | 테스트, 사전 워밍업, RI 전용 노드 |

---

## 4.2 자동확장 수량 조정 (Autoscale Limits)

![Auto-Scaling 수량 조정](images/node_scaling/1780271330270.png)

1. **Clusters → 클러스터 → Edit → Required Settings**.
2. **Max HPC Cores**(코어/인스턴스 상한) 수정.
3. **Save**.

> ⚠️ **버전별 Count 기준**: CycleCloud 8.6 이전은 **CPU 코어 수**, 8.7 이후는 **VM 인스턴스 개수** 기준입니다.

> **실행 중 클러스터 반영**: UI로 노드 수를 늘리거나 파티션을 추가해도 `sinfo`만 정상으로 보이고 실제 할당이 실패할 수 있습니다(`Unknown node name(s): ...-hpc-4`). 증설은 스케줄러에서 `sudo azslurm scale`(4.6)로 `azure.conf`를 재생성해 반영하고, 재시작 후 유지되도록 UI 값도 저장합니다.

### 축소 (`azure.conf` 직접 편집, 재시작 없이)

감설은 스케줄러의 `azure.conf` 노드 범위를 줄여 즉시 반영합니다.

```bash
cyclecloud connect scheduler -c slurm-first-cluster
sudo vi /sched/slurm-first-cluster/azure.conf   # 예: [1-3] → [1-2]
sudo scontrol reconfigure
```

> 수동 편집은 이후 `azslurm scale`/재시작 시 템플릿 기준으로 덮어써집니다(→ 4.6). 영구 반영은 UI/템플릿 값을 함께 수정하세요.

---

## 4.3 수동 노드 할당·회수 (`azslurm`)

스케줄러 노드에서 `root`로 실행합니다.

![sinfo 노드 상태 확인](images/node_scaling/1780366566685.png)

```bash
sinfo                                          # 노드 상태 확인 (idle~ = 정지)
sudo azslurm resume  --node-list hpc-1         # 기동 (Acquiring → Ready)
sudo azslurm suspend --node-list hpc-1         # 회수 (Deallocate/Delete)
```

---

## 4.4 Scale-in 방지: RI/GPU 노드 (`SuspendExcParts`)

Slurm은 기본 300초(5분) 유휴 노드를 자동 감설합니다. KT의 **RI(Reserved Instance) GPU VM**은 자동 감설 시 재할당 실패(Capacity 부족) 위험이 크므로 축소 대상에서 제외합니다. 포털 `keep_alive`는 오작동할 수 있어 Slurm 설정으로 제외합니다.

```ini
# /etc/slurm/slurm.conf
SuspendExcParts=hpc          # 파티션 전체 제외 (SuspendExcNodes=<노드>로 개별 제외도 가능)
```

```bash
sudo scontrol reconfigure
```

---

## 4.5 노드 사이즈(VM SKU) 변경

동작 중 VM 크기는 실시간 변경되지 않습니다. **SKU 변경 → 기존 노드 Terminate → 새 크기로 재생성** 순서입니다.

1. **Clusters → 클러스터 → Edit**.
2. 대상 배열의 **Machine Type** 수정(예: `Standard_D4s_v5` → `Standard_D8s_v5`).
3. **Save**.
4. **Arrays 탭**에서 기존 노드 **Terminate** → 다음 작업/`azslurm resume` 시 새 SKU로 생성.

> VM SKU 변경과 **OS 디스크 크기 변경**은 별개입니다. 디스크 크기는 [15. 디스크 사이즈 변경](15-디스크-사이즈-변경.md)을 참고하세요.

## 4.6 변경사항 적용 (`azslurm scale`)

포털 Edit/템플릿 변경 후 스케줄러에서 `azslurm scale`을 실행하면 **재시작 없이** 반영됩니다. `azure.conf`·`gres.conf`를 재생성하고 `slurmctld`를 재시작하며, 실행 중 작업에는 영향이 없습니다.

```bash
cyclecloud connect scheduler -c slurm-first-cluster
sudo -i
azslurm scale
```

`azslurm scale`은 **템플릿** 기준으로 재생성하므로 `azure.conf` 수동 편집분은 사라집니다. 영구 반영은 템플릿을 수정합니다.

```bash
cyclecloud export_parameters slurm-train > slurm-train.json
cyclecloud import_cluster slurm-train -c slurm -f slurm.txt -p slurm-train.json --force
sudo azslurm scale
```

## 4.7 `azslurm` 주요 명령 및 KeepAlive 주의

| 명령 | 용도 |
|------|------|
| `azslurm scale` | 템플릿 기준 `azure.conf` 재생성 + `slurmctld` 재시작 |
| `azslurm resume/suspend --node-list` | 노드 즉시 기동/회수 |
| `azslurm partitions` | 파티션 정의 출력 |
| `azslurm limits` | 쿼터 기준 가용 노드 수 |
| `azslurm keep_alive` | 특정 노드 축소 제외/해제 |
| `azslurm retry_failed_nodes` | 실패 노드 재시도 |
| `azslurm remove_nodes` | 인스턴스는 두고 스케줄러에서만 제거 |

> ⚠️ **KeepAlive/좀비 노드**: 포털 `KeepAlive=true`로도 Slurm 상태가 `powered_down`이 되어 좀비 노드(`down~`/`drained~`)가 될 수 있습니다. 축소 방지는 **4.4 `SuspendExcParts`** 또는 `azslurm keep_alive`를 쓰고, 좀비 노드는 `systemctl restart slurmd` 또는 `azslurm suspend`로 정리합니다.

## 4.8 노드 재부팅 SOP (용량/RI 안전 절차)

용량 부족 위험은 VM을 **deallocate(할당 해제)** 했다 다시 켤 때 발생합니다. **할당을 유지한 채 OS만 재부팅**하면 발생하지 않습니다.

> **Quota ≠ Capacity**: Quota(구독 vCPU 상한)는 늘릴 수 있어도, Capacity(리전 물리 재고)가 없으면 재할당이 실패합니다. RI는 요금 할인일 뿐 용량을 보장하지 않으며, 용량 보장은 **On-demand Capacity Reservation**이 필요합니다.

| 방법 | 동작 | 용량 위험 |
|------|------|-----------|
| `az vm restart -g <rg> -n <vm>` | 할당 유지 재부팅 | 없음 ✅ |
| 포털 → VM → **Restart** | 할당 유지 재부팅 | 없음 ✅ |
| 노드 내부 `sudo reboot` | OS 재부팅 | 없음 ✅ |
| 포털 **Stop(Deallocate)** → Start | 할당 해제 후 재획득 | 높음 ❌ |
| **Terminate** / `azslurm suspend` | 노드 삭제 후 재생성 | 높음 ❌ |

### in-place 재부팅 절차

```bash
scontrol update nodename=<node> state=drain reason="planned reboot"  # 새 작업 배치 중단
az vm restart -g <rg> -n <node>                                      # drain 완료 후 재부팅
scontrol update nodename=<node> state=resume                         # 복귀
```

### deallocate가 불가피한 경우 (예: SKU 변경 4.5)

1. **사전 quota 확인**: `az vm list-usage -l <region>`, 부족 시 증설 요청.
2. **On-demand Capacity Reservation**으로 SKU 용량을 선확보(deallocate 중에도 용량 홀드).
3. 확보가 어려우면 Azure Capa 팀과 일정 조율(→ [1장](01-환경-개요.md)).
