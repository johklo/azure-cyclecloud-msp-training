# 4. 노드 증설/감설 및 노드 사이즈 변경

CycleCloud Slurm 클러스터에서 HPC 노드의 수량과 크기를 조절하는 방법을 정리합니다.

---

## 4.1 Scale-in 방지

Slurm은 필요할 때만 자원을 할당하고, 사용하지 않으면 자동으로 회수하여 비용 효율적으로 운영합니다.

기본적으로 Suspend time이 **300초(5분)** 이므로, 노드에 Job이 할당되지 않으면 5분 내에 Scale-in 됩니다.

하지만 **Reserved Instance(RI) 계약**이 포함되어 Scale-in이 불필요한 경우에는 자동 축소를 방지해야 합니다. RI/GPU 노드는 자동 회수 후 재할당 시 용량 부족으로 실패할 위험이 크기 때문입니다.

> ⚠️ CycleCloud UI의 `keep_alive` 옵션은 정상적으로 작동하지 않습니다. (2025년 9월 기준) 따라서 아래 Slurm 설정으로 제외합니다.

파티션별 Scale-in 제외는 스케줄러 노드의 `/etc/slurm/slurm.conf` 에서 설정합니다.

```ini
# /etc/slurm/slurm.conf
SuspendExcParts=hpc      # hpc 파티션의 노드를 자동 축소 대상에서 제외
```

설정을 반영합니다.

```bash
sudo scontrol reconfigure
```

> 개별 노드만 제외하려면 `SuspendExcNodes=<노드명>` 을 사용합니다.

## 4.2 파티션별 최대 수량 조정

노드 **증설**은 아래 수량 조정을 먼저 진행하고, 노드 **감설**은 수량 조정 뒤 [4.3](#43-노드-할당-및-회수)의 회수를 진행합니다.

> - 이미 동작 중인 클러스터에서는 UI로 조정해도 즉시 반영되지 않습니다. 다만 나중에 있을 재시작을 고려하여 **UI와 스케줄러 양쪽 모두** 반영하는 것을 권장합니다.
> - CycleCloud 8.6 이전에는 count 값이 **코어 수** 기준이었으나, 8.7+ 부터는 **인스턴스 수** 기준입니다.

### 1) CycleCloud UI에서 수량 조정

확장하거나 새 파티션을 추가할 경우, `sinfo` 로는 정상으로 보이지만 실제 반영은 되지 않을 수 있습니다.

예를 들어 노드 범위를 `[1-3]` 에서 `[1-4]` 로 늘려도, 실제로는 아래처럼 실패합니다.

```
Error 'Unknown node name(s): cycle-lab-hpc-4'
```

> **Clusters → 클러스터 → Edit → Required Settings → Auto-Scaling** 에서 최대 수량을 조정한 뒤 **Save**.

![Auto-Scaling 수량 조정](images/node_scaling/1780271330270.png)

저장 후 스케줄러에서 `sudo azslurm scale` 을 실행하면 `azure.conf` 가 재생성되어 재시작 없이 반영됩니다.

### 2) 스케줄러 노드에서 수량 조정 (축소 전용)

감설은 스케줄러의 `azure.conf` 노드 범위를 직접 줄여 즉시 반영할 수 있습니다.

1. 스케줄러 노드에 SSH 접속합니다.

   ```bash
   cyclecloud connect scheduler -c <클러스터명>
   ```

2. `azure.conf` 를 수정해 노드 범위를 줄입니다. 예: `[1-3]` → `[1-2]` (최대 3대 → 2대)

   ```bash
   sudo vi /sched/<클러스터명>/azure.conf
   ```

   ```ini
   PartitionName=slurm-lab1-hpc Nodes=slurm-lab1-hpc-[1-2] Default=YES DefMemPerCPU=1536 MaxTime=INFINITE State=UP
   Nodename=slurm-lab1-hpc-[1-2] Feature=cloud STATE=CLOUD CPUs=2 ThreadsPerCore=1 RealMemory=3072
   ```

3. 설정을 반영합니다.

   ```bash
   sudo scontrol reconfigure
   ```

> 수동 편집분은 이후 `azslurm scale` / 클러스터 재시작 시 템플릿 기준으로 덮어써집니다. 영구 반영은 UI 값도 함께 수정하세요.

## 4.3 노드 할당 및 회수

`azslurm` 명령으로 노드를 즉시 할당하거나 회수할 수 있습니다. 스케줄러 노드에서 `root` 로 실행합니다.

현재 상태를 확인합니다. (`idle~` 는 정지 상태)

```bash
sinfo
```

![sinfo 노드 상태 확인](images/node_scaling/1780366566685.png)

노드를 할당(기동)합니다.

```bash
sudo -i
azslurm resume --node-list cyclecloud-lab-hpc-1
```

노드를 회수(삭제)합니다.

```bash
azslurm suspend --node-list cyclecloud-lab-hpc-1
```

## 4.4 노드 사이즈(VM SKU) 변경

동작 중인 VM 크기는 실시간으로 변경되지 않습니다. **SKU 변경 → 기존 노드 Terminate → 새 크기로 재생성** 순서로 진행합니다.

1. **Clusters → 클러스터 → Edit** 이동.

2. 대상 배열의 **Machine Type** 을 변경합니다. (예: `Standard_D4s_v5` → `Standard_D8s_v5`)

3. **Save** 클릭.

4. **Arrays 탭**에서 기존 노드를 **Terminate** → 다음 작업 제출 또는 `azslurm resume` 시 새 SKU로 생성됩니다.

> VM SKU 변경과 **OS 디스크 크기 변경**은 별개입니다. 디스크 크기는 [15. 디스크 사이즈 변경](15-디스크-사이즈-변경.md)을 참고하세요.

## 4.5 노드 재부팅 SOP (용량/RI 안전 절차)

용량 부족 위험은 VM을 **deallocate(할당 해제)** 했다가 다시 켤 때 발생합니다. **할당을 유지한 채 OS만 재부팅**하면 발생하지 않습니다.

> **Quota ≠ Capacity**: Quota(구독 vCPU 상한)를 늘려도 Capacity(리전 물리 재고)가 없으면 재할당이 실패합니다. RI는 요금 할인일 뿐 용량을 보장하지 않으며, 용량 보장은 **On-demand Capacity Reservation** 이 필요합니다.

| 방법 | 동작 | 용량 위험 |
|------|------|-----------|
| `az vm restart -g <rg> -n <vm>` | 할당 유지 재부팅 | 없음 ✅ |
| 포털 → VM → **Restart** | 할당 유지 재부팅 | 없음 ✅ |
| 노드 내부 `sudo reboot` | OS 재부팅 | 없음 ✅ |
| 포털 **Stop(Deallocate)** → Start | 할당 해제 후 재획득 | 높음 ❌ |
| **Terminate** / `azslurm suspend` | 노드 삭제 후 재생성 | 높음 ❌ |

in-place 재부팅 절차는 다음과 같습니다.

```bash
scontrol update nodename=<node> state=drain reason="planned reboot"   # 새 작업 배치 중단
az vm restart -g <rg> -n <node>                                       # drain 완료 후 재부팅
scontrol update nodename=<node> state=resume                          # 복귀
```
